Summary

The reviewed code — the app shell and the SwiftUI-adjacent AppKit UI framework — is small, clean, and internally consistent. Naming, formatting, licensing headers, and the Cocoa-bindings architecture (array controller + value transformers + KVO-driven SensorHistoryData) are all coherent, and the underlying SensorsKit data layer is carefully documented and thread-safe. No critical or high-severity defects were found in the reviewed scope.

The most significant issue is a latent crash in the graph rendering: the "bars" style computes a column count by dividing by a height-derived value and converting the result to Int, which traps when the view is drawn at zero height (Int(infinity) / Int(nan)). This is the same class of scroll-time drawing crash that commit f154b91 patched for the line/gradient path, but the bars and background paths were never guarded. The remaining items are a never-removed notification observer, a couple of edge-case rendering glitches, and minor style/robustness nits.

Findings

1. Graph "bars"/background rendering can trap on Int(infinity) / Int(nan) at zero height

Medium crash / stability

SensorsUI/SensorGraphView.swift:136–141 (background), SensorsUI/SensorGraphView.swift:224–230 (bars)

Description Both the bars background and the bars foreground derive a column count from the view height and then convert it to an Int for a range loop:

let c = 15
let h = rect.size.height / CGFloat( c )   // 0 when the view height is 0
let n = rect.size.width / h                // width / 0  ->  +infinity  (or 0/0 -> NaN)

for i in 0 ..< Int( n )                    // Int(+infinity) / Int(NaN)  ->  fatal trap

In Swift, converting a non-finite Double/CGFloat to Int with the Int(_:) initializer is a hard runtime trap ("… cannot be converted to Int because it is either infinite or NaN"). When rect.size.height == 0, h is 0; if the width is non-zero the division yields +infinity, and if the width is also zero it yields NaN — either way Int(n) crashes the app.

This is reachable: draw(_:) uses self.bounds, and collection-view items are laid out and redrawn during scrolling and resize, where a transient zero-height geometry is plausible. This is exactly the scenario behind commit f154b91 ("Fixed a crash when scrolling in the sensors view"), but that fix only added a NaN guard to the line/gradient path in drawGraph (lines 184–187). The bars path and the background path have no equivalent guard, so in the bars style the crash class remains open.

Note: the default style is gradient (stored default 0), so this only bites users who have selected the bars style — which is why it likely hasn't been reported.

Recommendation Guard against a degenerate rect before dividing, e.g. early-return from draw(_:) / the bars helpers when rect.size.height or rect.size.width is not positive, or use the failable Int(exactly:) / clamp n to a finite non-negative value. A single guard rect.width > 0, rect.height > 0 else { return } at the top of draw(_:) covers all drawing paths uniformly.

2. UserDefaults.didChangeNotification observer is never removed

Low stability / lifecycle

SensorsUI/SensorGraphView.swift:47, 53–57, 63–67 (no deinit)

Description Both initializers register a block-based observer and stash the opaque token in defaultsObserver, but the class has no deinit and never calls NotificationCenter.default.removeObserver(_:). Block-based observers registered with addObserver(forName:object:queue:using:) must be explicitly removed; otherwise the notification center retains the token and the block for the lifetime of the process. The [weak self] capture correctly prevents a retain cycle and prevents use-after-free, so this is not a crash — but every SensorGraphView ever instantiated keeps observing forever, and each fires on every app-wide defaults write (including unrelated writes such as the filter toggles and lastStart), redundantly invalidating the view. The leak is bounded by the collection view's reuse pool, so impact is modest.

Recommendation Add a deinit that removes the stored token: if let o = self.defaultsObserver { NotificationCenter.default.removeObserver(o) }. Optionally narrow the trigger by observing only the relevant key rather than redrawing on any defaults change.

3. A constant sensor renders as full bars (NaN not handled in the bars path)

Low correctness / UI

SensorsUI/SensorGraphView.swift:238, 242

Description In drawBars, the fill fraction is let p = ( v - min ) / ( max - min ). When every sample in the window is equal (a perfectly flat sensor — common for idle voltages/fans), min == max, so this is 0.0 / 0.0 == NaN. The break condition CGFloat( j ) / CGFloat( c ) > p is always false for a NaN right-hand side, so the inner loop never breaks and draws all 15 squares — i.e. a flat reading is displayed as a maxed-out bar column. The line/gradient path avoids this by special-casing min == max (lines 167–173), but the bars path has no such guard.

Recommendation Handle max == min explicitly in drawBars (e.g. render a single baseline/mid row, mirroring drawGraph), or clamp p to 0 when the range is zero.

4. Malformed first-parameter name in the collection-view data-source method

Low style / maintainability

SensorsUI/SensorsWindowController.swift:214

Description The method is declared as:

public func collectionView( _ itemForRepresentedObjectAtcollectionView: NSCollectionView,
                            itemForRepresentedObjectAt indexPath: IndexPath ) -> NSCollectionViewItem

The first parameter's internal name is a copy/paste mash-up (itemForRepresentedObjectAtcollectionView). This is not a functional bug: the Objective-C selector is still collectionView:itemForRepresentedObjectAt: (the internal name doesn't affect it), the data source is wired to File's Owner in the XIB, and the body uses the self.collectionView outlet rather than the passed argument — so it works. But the signature reads as an accidental rename, is confusing, and is fragile against future edits.

Recommendation Restore the conventional signature collectionView( _ collectionView: NSCollectionView, itemForRepresentedObjectAt indexPath: IndexPath ) and use the passed-in collectionView parameter.

5. Force-unwrapped updater IBOutlet

Info style / convention

Sensors/Classes/ApplicationDelegate.swift:35

Description @IBOutlet private var updater: GitHubUpdater! is an implicitly-unwrapped optional, dereferenced in applicationDidFinishLaunching and checkForUpdates. If the outlet is ever disconnected in the XIB, this traps. The project's own coding conventions call out avoiding force-unwrap even for IBOutlets. Practically low-risk since the outlet is wired in MainMenu.xib, but flagged for consistency with the stated convention.

Recommendation Prefer a weak optional outlet with graceful handling (self.updater?.checkForUpdatesInBackground()), or document the deliberate exception.

6. SensorValue uses a fixed, non-localized %.02f for every kind

Info UI / formatting

SensorsUI/SensorValue.swift:48

Description All sensor values are formatted with String(format: "%.02f", …). This is fine for temperatures/volts/amps, but the same two-decimal, non-localized format is applied to fan speed (RPM) and ambient light — where fractional digits are meaningless (e.g. 2100.00 RPM) — and it ignores the user's locale decimal separator. Since the transformer receives the value only (no kind), it can't adapt per kind today.

Recommendation Consider a NumberFormatter (locale-aware) and/or per-kind precision; RPM in particular reads better as an integer. Low priority / cosmetic.

7. Inconsistent default for the graph style fallback

Info robustness

SensorsUI/SensorGraphView.swift:72

Description GraphStyle( rawValue: UserDefaults.standard.integer( forKey: "sensorsWindowGraphStyle" ) ) ?? .bars. The stored default for an unset key is 0, which is .gradient, so a fresh install renders gradients — but the failure fallback (for an out-of-range stored value) is .bars. The two "defaults" disagree, and given finding 1, silently falling back to .bars steers toward the more crash-prone path. Minor.

Recommendation Make the fallback match the intended default (?? .gradient) for consistency.

8. No unit tests for the Sensors / SensorsUI targets

Info test coverage

Repository-wide (Sensors/, SensorsUI/)

Description The SensorsKit submodule has a solid test suite, but the app and UI-framework targets in this repository have none. The pure, side-effect-free logic here is testable in isolation — e.g. the graph min/max/NaN handling, the filter-predicate assembly in updateFilters, and the value transformers. The rendering-crash class in finding 1 is precisely the kind of thing a small unit test over the geometry math would pin down.

Recommendation Add focused tests for the value transformers and the graph geometry helpers (extracting the pure math from draw if needed).

Strengths

  • Consistent, readable style throughout — aligned assignments, early returns, one type per file, uniform license headers, and clear naming.
  • Careful concurrency in the data layer — SensorHistoryData serializes buffer access via Synchronizable, publishes on the main queue, and uses [weak self] in the polling thread/timer so the object can be torn down by deinit without leaking the run loop.
  • Deterministic ordering — the fixed en_US locale plus a total-order tie-break in Sensors.sorted keeps rows from shuffling between polling passes.
  • Prior crash awareness — the NaN guard added in f154b91 shows the drawing edge cases are understood; finding 1 is simply the remaining, unguarded sibling of that same fix.
  • Clean bindings architecture — array controller, value transformers, and KVO-observable model keep the controllers thin.
